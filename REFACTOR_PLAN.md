# 重构计划与分析：短链接批量写入服务

**日期:** 2025-12-08
**主题:** 访问日志与统计数据批量处理的优化

本文档记录了短链接项目（slink-api）中 `BatchWriterService` 及核心模块的重构分析、问答记录及路线图。

---

## 1. 核心问题：统计数据写入性能（高优先级）

### 用户疑问 / 关切
> “如果是改成读取到内存再分批写入，是不是意味着只有攒够了分批的大小（比如500条）才会入库？如果流量很小，数据会不会一直卡在内存里？我担心时效性问题。”

### 分析
*   **当前实现：** 定时任务每 10 秒运行一次。它通过 `SCAN` 遍历 Redis Key。每发现 *一个* Key，就立即执行一次数据库 `UPDATE/INSERT` 操作。
    *   *性能影响：* 100 个 Key = 100 次数据库事务。速度极慢，连接开销巨大。
*   **重构方案：** 定时任务仍然每 10 秒运行一次。它遍历 Key 并将数据追加到内存切片（Slice）中。
    *   *触发条件 A（满载）：* 如果切片长度 >= 500，立即执行一次 SQL 批量插入，然后清空切片。
    *   *触发条件 B（收尾）：* 当所有 Redis Key 扫描完毕后，如果切片长度 > 0（例如只有 5 条），立即执行一次 SQL 批量插入。
*   **结论：** 数据延迟仅取决于 Cron 的间隔（10秒），而**不取决于**批次大小。低流量场景由“触发条件 B”处理，确保数据不会滞留。

### 解决方案计划
1.  修改 `internal/service/writer.go` 中的 `syncStatsCounters` 方法。
2.  为每种统计类型（`StatsDaily`, `StatsRegion`, 等）引入累加器（切片）。
3.  实现从 Redis 收集数据的循环逻辑。
4.  仅当累加器满或循环结束时，才执行 `db.CreateInBatches` 或 `clauses.OnConflict`。
5.  **收益：** 将数据库 I/O 调用次数从 N 降低到 N/500。

## 2. 问题：Redis `KEYS` vs `SCAN`（性能风险）

### 现状
*   **原始代码：** 可能使用了 `KEYS`（阻塞性命令）。
*   **当前代码：** `internal/service/writer.go` 已经使用了 `SCAN`。
*   **优化：** 虽然 `SCAN` 是非阻塞的，但逐个处理 Key 仍然低效。问题 #1（批处理）中的解决方案通过降低每个扫描项的后续处理成本，自然地补充了 `SCAN` 的优势。

## 3. 问题：数据安全与崩溃恢复

### 分析
*   **机制：** 系统使用 `Rename` 将数据从“活跃 Key”移动到“处理中 Key”。
*   **风险：** 如果应用程序在 `Rename` 之后、数据库写入/恢复 *之前* 崩溃（Panic/OOM）：
    *   “处理中 Key”存在于 Redis 中，但变成了“孤儿数据”。
    *   除非有额外的手动清理脚本查找 `*:processing:*` Key，否则这些统计数据实际上就丢失了。
*   **缓解措施（未来工作）：**
    *   实现“启动恢复”或单独的“清理 Cron”，扫描过期的 `processing` Key（如超过 10 分钟）并重新合并它们。
    *   *注：* 这不属于当前的立即重构范围，但在健壮性方面值得注意。

## 4. 问题：事务完整性（日志 vs 统计）

### 分析
*   **场景：** 访问日志（Access Log）和统计数据（Statistics）存储在不同的表中（且逻辑上属于不同的数据流）。
*   **问题：** 可能出现 `syncRawLogs` 成功（日志已保存）但 `syncStatsCounters` 失败（统计未更新）的情况，反之亦然。
*   **决策：** 我们接受 **最终一致性（Eventual Consistency）**。鉴于这是一个分析系统，在 Redis 到 MySQL 的同步过程中追求严格的 ACID 事务代价过高。
*   **策略：** 专注于“至少一次（At-least-once）”投递。补偿机制（`restoreRawLogs`）试图确保数据不丢失，即使这可能导致罕见的重复计数（`ON DUPLICATE KEY UPDATE` 能优雅地处理统计数据的重复，尽管原始日志如果 ID 不去重可能会有重复）。
